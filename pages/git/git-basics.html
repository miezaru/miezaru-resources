Git

Всё что попало в коммит почти всегда Git может восстановить. Можно восстановить даже коммиты из веток, которые были удалены, или коммиты, перезаписанные параметром --amend (см. Восстановление данных). Но всё, что не было включено в коммит и потеряно — скорее всего, потеряно навсегда.

Git: как устроен Git

VCS - Version Control System

Version Control System (VCS) - это система, записывающая изменения в файл или набор файлов в течение времени и позволяющая вернуться позже к определённой версии.

Раньше были локальные системы контроля версий (RCS), но что-бы доступ был к нескольким разработчикам, а не одному то придумали централизованные системы контроля версий (CVCS). К ним относились такие системы, как CVS, Subversion, Perforce. Пользовались большим спросом много лет.

Сейчас все пришло к использованию распределенных систем контроля версий (DVCS), ибо если с диском или сервером предыдущих вариантов что-то случится и нету своевременных обновленных бекапов, то все данные будут утеряны. В DVCS каждая копия репозитория является полным бэкапом всех данных.

Линейные VCS используются и сегодня довольно таки активно из-за своей простоты и не наобходимости в DAG - Directed acyclic graph ( Направленный ациклический граф).

VCS Differences

  1. Системы VSC по типу CVS, Subversion, Perforce, Bazaar хранят изменения в виде списков.

  2. Git хранит изменения в виде снимков (snapshot).

  3. Большой плюс Git - работы без интернета, при помощи коммитов и сохранение изменений при появлении интернета с помощью команды push.

Git stages

Три состояния фалов в git репозитории:

  1. Изменён (modified)

    • К изменённым относятся файлы, которые поменялись, но ещё не были зафиксированы.

  2. Индексирован (staged)

    • Индексированный — это изменённый файл в его текущей версии, отмеченный для включения в следующий коммит.

  3. Зафиксирован (committed)

    • Зафиксированный значит, что файл уже сохранён в вашей локальной базе.

Git: базовые комманды

Создание репозитория:

  1. git init

  2. git add *.html

  3. git add License

  4. git commit -m "Initial commit"

Клонирование репозитория:

  1. git clone https://github.com/url [optional: каталог]

  2. git clone https://github.com/libgit2/libgit2 mylibgit

Управление файлами в репозитории:

  1. git status -s

    • короткий вывод изменений

  2. git diff

    • просмотр конкретных измененных строк в файлах

  3. git diff --staged

    • что войдет в следующий коммит (сравнивает проиндексированные изменения с последним коммитом)

  4. git add file.html

    • индексация файла file.html

  5. git restore --staged file.html

    • убирает файл с проиндексированных

  6. git restore file.html

    • опасная команда. Заменяет все локальные изменения в файле версией с последнего коммита.

  7. git checkout -- file.html

    • опасная команда. Заменяет все локальные изменения в файле версией с последнего коммита.

Git: Коммиты

  1. git commit -m "comment"

    • снимок проиндексированных изменений

  2. git commit -a -m "comment"

    • короткая версия записи коммита с добавление индексации (git add .)

  3. git commit --amend

    • замена предыдущего коммита новый с новыми изменениями

    Git: Удаление файлов

  1. git rm file.html

    • удаление файла из отслеживаеммых

  2. git rm -f file.html

    • удаление файла если его уже изменили и проиндексировалли

  3. git rm --cached file.html

    • удалить файл, но оставить его в каталоге

  4. git rm log/\*.log

    • удаление всех файлов в папке log с окончание .log

  5. git rm \*.log

    • удаление всех файлов с окончание .log

    Git: Переименование файлов

  1. git mv README.md README

    • переименование файлов (это лиш укороченная версия ввода трех команд):

    • mv README.md README

    • git rm README.md

    • git add README

Просмотр коммитов:

У меня уже есть настроенные git lg1, git lg2

Можно настроить собственный более красивый вывод: git log

Удаленные репозитории

  1. git remote

    • названия доступных репозиториев

  2. git remote -v

    • выведет адресса чтения и записи (возможно несколько адрессов, при работе с несколькими разработчиками)

    • origin - название внешнего git сервера по умолчанию командой git clone.

  3. git clone -o server-name

    • изменение стандартного имени сервера "origin" на "server-name"

  4. git remote add john https://github.com/john/our-project

    • добавление удаленных репозиториев

  5. git fetch john

    • получить изменения Джона, не изменяет наши изменения.

  6. git pull

    • git fetch + git merge (с сервером на который смотритрит репозиторий (origin))

  7. git push <remote-name> <branch-name>

    • отправка изменений в удаленный репозиторий

  8. git push origin master

    • отправка изменений в удаленный репозиторий (не позволит отправить если сервер origin изменился с последнего git pull)

  9. git remote show origin

    • получить больше информации об удаленном репозитории, в данном примере origin

  10. git remote rename john jn

    • переименование удаленного репозиторий

  11. git remote rm john

    • удаление удаленного репозитория

Git: теги/tags

Работа с тегами

  1. git tag

    • список тегов

  2. git tag -l "v1.8.5*"

    • поиск тегов по шаблону

  3. git tag -a v1.4 -m "my version 1.4"

    • аннотированный тег (рекомендуется, ибо хранит больше информации)

  4. git tag v1.4

    • легковестный тег, хранит только ссылку на контрольную сумму коммита

  5. git log --pretty=oneline

    • список пройденных коммитов

  6. git tag -a v1.2 9fceb02

    • отложенно задать тег на коммит

Обмен тегами

  1. git push origin <tagname>

    • git push не отправляет теги на удаленные сервера, их надо отправлять явно.

  2. git push origin --tags

    • отправить все теги, которых нету на сервере origin

  3. git push <remote> --follow-tags

    • отправить только аннотированные теги

Удаление тегов

  1. git tag -d v1.4-lw

    • удаляет тег v1.4-lw локально

  2. git push origin --delete v1.4-lw

    • удаляет тег v1.4-lw на внешнем сервере

  3. git push origin :refs/tags/v1.4-lw

    • это следует понимать как обновление внешнего тега пустым значением, что приводит к его удалению. Еще один способ удаления тега на внешнем сервере.

Переход на тег

  1. git checkout v2.0.0

    • однако, это переведёт репозиторий в состояние «detached HEAD», которое имеет ряд неприятных побочных эффектов.

    • Если в состоянии «detached HEAD» внести изменения и сделать коммит, то тег не изменится, при этом новый коммит не будет относиться ни к какой из веток, а доступ к нему можно будет получить только по его хешу. Поэтому, если вам нужно внести изменения — исправить ошибку в одной из старых версий — скорее всего вам следует создать ветку:

  2. git checkout -b version2 v2.0.0

    • создание и переключение на ветку version2

    • если сделать коммит в ветке version2, то она сдвинется вперед и будет отличаться от тега v2.0.0, так что будьте с этим осторожны.

Git: ветвление

Git branch basics

  1. git branch testing

    • создание ветки testing (новый указатель на текущий коммит)

    • HEAD - указатель на текущую локальную ветку

  2. git log

    • покажет куда смотрит HEAD

    • git log не показывает все ветки по умолчанию, для просмотра истории коммитов другой ветки необходимо явно указать её имя:

  3. git log testing

    • явно посмотреть логи ветки testing

  4. git log --all

    • посмотреть историю по всем веткам

  5. git checkout testing

    • переключение на ветку testing (что-бы HEAD смотрел теперь на нее) (old)

  6. git switch testing

    • переключение на ветку testing (что-бы HEAD смотрел теперь на нее) (new)

  7. git checkout 18b9746

    • есть возможность прыгать на определенный коммит, и уже с него создавать доп ветки

  8. git checkout -b <branchname>

    • создание и сразу переключение на новую ветку (old)

  9. git switch -c/--create <branchname>

    • создание и сразу переключение на новую ветку (new)

  10. git switch -

    • вернуться к предыдущей извлеченной ветке

Слияние веток

  1. git merge <branchname>

    • слияние веток. Проводить с ветки на которую мы хотим произвести слияние <branchname>. Если <branchname> является прямым потомком текущей ветки, то происходит fast-forward. Git просто переносит указатель ветки вперед, так как расхождений в изменениях нету.

После слияния ветку можно удалить.

  1. git branch -d <branchname>

    • удалить ветку branchname

Возможны проблемы со слияниями, если изменения в ветках затронули одни и теже файлы, в таком случае прийдется все починить в ручную либо выбрать в какой из веток вариант правильный.

После завершения ручных изменений в конфликтах, необходимо просто прописать git add <file> , что-бы обозначить проблему решенной и закоммитить git commit

Управление ветками

  1. git branch

    • список веток (* - показывает на какую ветку указывает HEAD)

  2. git branch -v

    • показывает последний коммит на каждой из веток

  3. git branch --merged

    • список веток слитых в текущую

  4. git branch --no-merged

    • список не слитых веток в текущую

  5. git branch --no-merged testing

    • расширенный вариант записи, что-бы глянуть информацию по другой ветке

  6. git branch -D <branchname>

    • что-бы удалить ветку без слияния вместе со всеми наработками

Переименование веток

Осторожно, особенно если менять название главной ветки, ибо скрипты и утилиты могут использовать старое имя главной ветки. Необходимо все тчательно проверять перед заменой

  1. git branch --move old-branch-name new-branch-name

    • переименование ветки локально

  2. git branch --set-upstream origin new-branch-name

    • изменение имени локальной ветки на удаленном репозитории

  3. git push origin --delete old-branch-name

    • удалить старую ветку на удаленном репозитории

Для замены главной ветки, просмотреть пособие на git book

Работа с ветками

Многие разработчики, использующие Git, придерживаются именно такого подхода, оставляя полностью стабильный код только в ветке master — возможно, только тот код, который был или будет выпущен. При этом существует и параллельная ветка с именем develop или next, предназначенная для работы и тестирования стабильности; она не обязательно должна быть всегда стабильной, но при достижении стабильного состояния её содержимое можно слить в ветку master.

Число уровней стабильности можно увеличить. В крупных проектах зачастую появляется ветка proposed или pu (предлагаемые обновления), объединяющая ветки с содержимым, которое ещё не готово к включению в ветки next или master. Идея состоит в том, что каждая ветка представляет собой определённый уровень стабильности; как только он повышается, содержимое сливается в ветку уровнем выше.

Тематические ветки

Тематической веткой называется временная ветка, создаваемая и используемая для работы над конкретной функциональной возможностью или решения сопутствующих задач.

Удаленные ветки

Удалённые ссылки — это ссылки (указатели) в ваших удалённых репозиториях, включая ветки, теги и так далее.

  1. git ls-remote <remote>

    • получение удаленных веток и дополнительной информации.

  2. git remote show <remote>

    • получение удаленных веток и дополнительной информации.

Ветки слежения

Ветки слежения — это ссылки на определённое состояние удалённых веток. Это локальные ветки, которые нельзя перемещать; Git перемещает их автоматически при любой коммуникации с удалённым репозиторием, чтобы гарантировать точное соответствие с ним.

Представляйте их как закладки для напоминания о том, где ветки в удалённых репозиториях находились во время последнего подключения к ним.

  1. git fetch <remote>

    • эта команда определяет какому серверу соответствует <remote> (origin), извлекает оттуда данные, которых у нас ещё нет, и обновляет локальную базу данных, сдвигая указатель origin/master на новую позицию.

  2. git remote add

    • добавлять внешнюю ветку

  3. git fetch <remote>

    • следить за внешней веткой локально

Поделиться веткой

Когда вы хотите поделиться веткой, вам необходимо отправить её на удалённый сервер, где у вас есть права на запись. Ваши локальные ветки автоматически не синхронизируются с удалёнными при отправке — вам нужно явно указать те ветки, которые вы хотите отправить.

Таким образом, вы можете использовать свои личные ветки для работы, которую не хотите показывать, а отправлять только те тематические ветки, над которыми вы хотите работать с кем-то совместно.

  1. git push <remote> <branch>

    • отправка указанной ветки на указанный сервер

  2. git checkout -b serverfix origin/serverfix

    • создание локальной ветки на основе original/serverfix, и которая начинается там же где и original/serverfix

Отслеживание веток

Получение локальной ветки из удалённой ветки автоматически создаёт то, что называется «веткой слежения» (а ветка, за которой следит локальная называется «upstream branch»).

Ветки слежения — это локальные ветки, которые напрямую связаны с удалённой веткой.

Если, находясь на ветке слежения, выполнить git pull, то Git уже будет знать с какого сервера получать данные и какую ветку использовать для слияния.

  1. git checkout -b <branch> <branch>/<remote>

    • новая ветка отслеживает удаленную

  2. git checkout --track origin/serverfix

    • тоже самое

  3. git checkout serverfix

    • тоже самое

Чтобы создать локальную ветку с именем, отличным от имени удалённой ветки, просто укажи другое имя:

  1. git checkout -b sf original/serverfix

    • новая ветка с именем sf

  2. git branch -vv

    • показывает какая из веток отслеживается, отстаёт, опережает или всё сразу относительно отслеживаемой

Важно отметить, что эти цифры описывают состояние на момент последнего получения данных с каждого из серверов. Эта команда не обращается к серверам, а лишь говорит вам о том, какая информация с этих серверов сохранена в локальном кэше.

Если вы хотите иметь актуальную информацию об этих числах, вам необходимо получить данные со всех ваших удалённых серверов перед запуском команды. Сделать это можно вот так:

  1. git fetch --all; git branch -vv

Получение изменений

Команда git fetch получает с сервера все изменения, которых у вас ещё нет, но не будет изменять состояние вашей рабочей копии. Эта команда просто получает данные и позволяет вам самостоятельно сделать слияние.

Тем не менее, существует команда git pull, которая в большинстве случаев является командой git fetch, за которой непосредственно следует команда git merge. Если у вас настроена ветка слежения как показано в предыдущем разделе, или она явно установлена, или она была создана автоматически командами clone или checkout, git pull определит сервер и ветку, за которыми следит ваша текущая ветка, получит данные с этого сервера и затем попытается слить удалённую ветку. Обычно, лучше явно использовать команды fetch и merge, поскольку магия git pull может часто сбивать с толку.

Удаление веток на удалённом сервере

  1. git push origin --delete serverfix

    • удаление удаленной ветки serverfix

    • всё, что делает эта строка — удаляет указатель на сервере. Как правило, Git сервер хранит данные пока не запустится сборщик мусора, поэтому если ветка была удалена случайно, чаще всего её легко восстановить.
